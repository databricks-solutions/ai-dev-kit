# MLflow Traces and Assessments Reference

## Trace Structure

A trace represents a single request through a GenAI application (agent, RAG pipeline, LLM chain).

### Trace Info

| Field | Description |
|-------|-------------|
| `trace_id` | Unique ID (e.g. "tr-abc123...") |
| `experiment_id` | Parent experiment |
| `state` | OK, ERROR, IN_PROGRESS |
| `request_time` | When the trace started (ISO 8601) |
| `execution_duration` | Duration (e.g. "16.357s") |
| `request_preview` | Truncated input |
| `response_preview` | Truncated output |
| `tags` | User-defined key-value metadata |
| `metadata` | System metadata (token usage, size, git info) |
| `assessments` | Feedback and expectations attached to the trace |

### Spans

Spans represent individual operations within a trace (LLM calls, tool invocations, retrieval steps). Each span has:

| Field | Description |
|-------|-------------|
| `span_id` | Unique span ID |
| `name` | Operation name (e.g. "predict", "retrieve_docs") |
| `parent_span_id` | Parent span (null for root) |
| `start_time_ns` / `end_time_ns` | Timing in nanoseconds |
| `status` | Status code and message |
| `attributes` | Span-specific attributes (model name, token counts) |

## Trace Metadata

The `metadata` dict contains system-generated fields:

| Key | Description |
|-----|-------------|
| `mlflow.trace.sizeBytes` | Total trace size |
| `mlflow.trace.tokenUsage` | JSON string with input_tokens, output_tokens, total_tokens |
| `mlflow.source.git.repoURL` | Git repo where the code was run |
| `mlflow.source.name` | Source file or notebook path |

### Token Usage

```python
# Get token usage from a trace
trace = get_mlflow_trace("tr-abc123")
import json
usage = json.loads(trace["metadata"].get("mlflow.trace.tokenUsage", "{}"))
print(f"Input: {usage.get('input_tokens')}, Output: {usage.get('output_tokens')}")
```

## Assessments

Assessments are annotations attached to traces â€” either human feedback or ground-truth expectations.

### Assessment Types

| Type | Purpose | Example |
|------|---------|---------|
| `feedback` | Human or LLM judge evaluation | "good", "bad", "yes", "no" |
| `expectation` | Ground-truth answer for evaluation | "The answer should mention X" |

### Source Types

| Source | Description |
|--------|-------------|
| `HUMAN` | Manually provided by a person |
| `LLM_JUDGE` | Automatically generated by an LLM evaluator |

### Assessment Workflow

```python
# 1. Search for traces to assess
traces = search_mlflow_traces(["123"], filter_string="status = 'OK'", max_results=20)

# 2. Inspect a trace
detail = get_mlflow_trace(traces["traces"][0]["trace_id"])

# 3. Log feedback
log_mlflow_assessment(
    detail["trace_id"],
    "quality",
    "good",
    rationale="Accurate response with correct tool usage"
)

# 4. Log expected answer for eval dataset
log_mlflow_assessment(
    detail["trace_id"],
    "expected_response",
    "The answer should list available products and prices",
    assessment_type="expectation"
)
```

### Built-in Assessment Names

Common assessment names used by MLflow evaluation:

| Name | Type | Values |
|------|------|--------|
| `retrieval_sufficiency` | feedback | "yes", "no" |
| `correctness` | feedback | "yes", "no" |
| `safety` | feedback | "safe", "unsafe" |
| `groundedness` | feedback | "grounded", "not_grounded" |
| `expected_response` | expectation | Free text |
| `expected_facts` | expectation | Free text |

## Trace Filtering

The `filter_string` parameter in `search_mlflow_traces` supports:

```
status = 'OK'
status = 'ERROR'
timestamp_ms > 1700000000000
```

## Tags vs Assessments

| Feature | Tags | Assessments |
|---------|------|-------------|
| Purpose | Metadata/labels | Quality judgments |
| Format | Key-value strings | Structured with source, rationale |
| Set by | `set_mlflow_trace_tag` | `log_mlflow_assessment` |
| Use case | Filtering, dataset building | Evaluation, feedback loops |
| Multiple values | One value per key | Multiple per assessment name |
